# The Filed Answer → Promote → Curate loop is human-governed *validated write-back*; promote auto-reindexes (full BM25 rebuild), filing skips qa-grounded answers, and the loop is deliberately not a semantic cache

The Filed Answer loop captures every grounding-passing `/chat` answer as a `status: draft` `wiki/qa/*.md` page; a curator reviews the Curation Queue and Promotes `draft → live`; only `live` pages enter the retrieval corpus ([[filed-answer]], Phase 6). A 2026-06-30 grill (issues #341/#342) traced a felt "prod feels fragile" bug to a **logic cycle**, not the ephemeral container: `promote` flipped `status` but **did not reindex**, so a promoted answer was never retrievable; and `dispatch_filing` re-fired on **every** grounded answer with no novelty guard, so re-asking the same question (the Gateway rewrites it on turn 2+, [[query-rewriting]]) filed a **new near-duplicate draft** under a different slug, which the Curation Queue (lint C8) then surfaced within seconds — making Promote look like a no-op. This ADR records the hardening decisions and, because the author needs to defend the architecture in an interview, the named industry pattern it implements and the simpler alternatives that were considered and rejected.

The loop is a **human-governed instance of "validated write-back" / safe self-improving RAG**: write a generated answer back into the corpus only after a multi-stage acceptance layer of *grounding verification* (our post-LLM [[grounding-check]]) + *attribution* (our [[citation]]s) + *novelty detection* (the new filing gate below), the same three stages the research literature names for safe corpus write-back (e.g. "Bidirectional RAG", arXiv 2512.22199), and the same shape every enterprise support-KB vendor ships as "resolved ticket → AI draft → human review → publish" (Zendesk, Aisera, InvGate, Supportbench). The one deliberate difference from the fully-automated research framing — a **human Promotion gate** instead of an automated confidence threshold — is the governance feature that makes this a *curated knowledge base* rather than an answer cache, and is the safe default for FAQ / policy / support content where a wrong published answer is a liability.

## Considered Options

- **Replace the loop with a semantic answer cache (rejected).** A semantic cache (GPTCache, RedisVL, vector-similarity query cache) inherently does the two things we hand-build — semantic dedup of questions and answer reuse — and is far simpler. Rejected as a *replacement* because a cache is a **latency/cost optimization, not a governance layer**: TTL-based freshness is insufficient for policy that changes, it has no human review, it is open to cache poisoning, and it cannot produce a grounded, citable, durable answer. It is **complementary** — a semantic cache could later sit *in front of* the stacks to cut latency on repeat queries — but it cannot be the system of record for an enterprise FAQ/policy KB. (We do borrow its embedding-similarity technique for the deferred semantic-dedup step.)
- **Fully-automated write-back / auto-publish (rejected for this use case).** Drop the human gate and let the acceptance layer publish directly (the literal "self-improving RAG" framing). Rejected because unreviewed write-back's central risk is **corpus pollution / hallucination accumulation** degrading future retrieval; for policy/support a human gate is the correct safe default. (Aisera offers exactly this as an opt-in "auto-publish" flow alongside its human-review flow — we choose the human-review flow.)
- **Incremental index update on promote (deferred, not rejected).** A true single-page incremental update is feasible — BM25's global statistics (document frequency, average length, N) update in O(new page) without re-tokenizing the corpus, and a vector add is natively incremental — and it is the production norm at scale. Deferred because at the current corpus (hundreds of pages) a full `build_index` rebuild is sub-second, **trivially correct** (the index is guaranteed to match disk, with no incremental-patch drift surface), and `index.json` is rewritten whole regardless; even production engines (Lucene/Elasticsearch) tolerate **approximate per-shard IDF** rather than recomputing global stats per add, so perfect incremental is not even the bar. Revisit when the corpus grows large enough that a single promote's rebuild is felt (order low-tens-of-thousands of documents, or a near-real-time update requirement) — at which point the right move is to adopt a backend with native incremental upsert (Elasticsearch/OpenSearch, pgvector/Qdrant), **not** to hand-roll an incremental BM25 patcher.
- **Rebuild the Hybrid dense index on promote (deferred to a separate issue).** `.kb/hybrid_dense` has **no production rebuild path at all** (`hybrid_kb` exposes no endpoint; only tests/eval build it), so the dense arm is frozen on the committed seed for all post-seed content — a pre-existing Phase 13 gap, not introduced here and not qa-specific. Not fixed in this slice because RRF fuses by Section id and tolerates a page present in only the BM25 arm (`reciprocal_rank_fusion` keys by `section.id`), so a promoted qa page is still served by the Hybrid stack via its BM25 arm; the dense arm merely adds no recall boost for it. Tracked as its own issue (favoured fix: an operator-triggered dense rebuild, symmetric with the existing `POST /rag/index`).

## Consequences

- **Promote auto-reindexes.** `POST /qa/{slug}/promote` flips `status` *and then* calls `build_index` (full BM25 rebuild of the Section Index). This deliberately departs from the documented "explicit `/index` is a separate step" model ([[section-index]]) because promote is a rare, single, intent-explicit curator action (≠ batch ingest); the explicit-`/index` model is retained for ingest. After promote, the Wiki stack and the Hybrid stack's BM25 arm retrieve the page immediately.
- **Filing gets a novelty gate.** `dispatch_filing` skips `maybe_file_answer` when **every** cited source is a `wiki/qa/` page (the answer is fully derived from already-curated Q&A → nothing new to capture). A re-ask whose rewritten/rephrased query misses the promoted qa page and is answered from raw Sources still files — accepted as a legitimate **coverage signal** (that phrasing isn't covered), to be collapsed later by semantic dedup.
- **Semantic dedup (heavier novelty detection) is deferred,** not dropped — it is a named stage of the validated-write-back pattern, and the qa-grounded gate above is already a cheap special case. When built it will cluster near-duplicate drafts (shared cited Sections + similarity, escalating to embedding similarity) and surface them in the Curation Queue for curator Discard ([[adr-0012]], read-only lint); auto-*merge* is explicitly out of scope (merging is a human judgement).
- **The Hybrid dense arm lags** for promoted/ingested content until its separate rebuild issue lands; this is runtime-safe (RRF tolerance) and degrades recall, not correctness.
- **This is a logic fix, independent of the ephemeral prod posture.** The cycle reproduces locally; the loop is designed for the persistent local-operator environment ([[operator-console]]). Whether the public `ask-wiki-rag` demo should persist runtime writes at all is a separate posture decision (issues #341 item 1 / #342), not addressed here.
- **Implementation** is a follow-up slice; this ADR records the design from the grill.
