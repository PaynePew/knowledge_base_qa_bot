# Hybrid Retrieval is a third stack — BM25 + dense over the wiki corpus, fused by RRF

Phase 13 adds **Hybrid Retrieval** as a third Retrieval Stack, in a new workspace package `hybrid_kb`, alongside Wiki (`markdown_kb`, BM25) and Vector RAG (`vector_rag`, dense over `docs/` Chunks). Hybrid runs **two retrieval methods over one corpus**: BM25 and a dense-vector index, **both over the curated `wiki/` Section corpus**, and fuses the two ranked lists with **Reciprocal Rank Fusion (RRF)**. It is **additive** — it does not modify the two existing stacks ([ADR-0002](0002-two-parallel-retrieval-apps.md)) — and is selectable everywhere the other two are (`stack=hybrid` on the Gateway, `--stack hybrid` on the CLI, `stack="hybrid"` on MCP), reusing the [ADR-0010](0010-gateway-mounts-both-apps.md) dispatch contract that already reserved `wiki|rag|hybrid`.

The dense arm is built **from `markdown_kb`'s already-filtered Section list** (entities + concepts + live qa), at **Section granularity** (not char-Chunk), so dense ids align **1:1** with BM25 Section ids and fusion is true same-corpus fusion. Because the retrieval unit stays `Section` (which satisfies the `CitableContent` protocol), every downstream concern — page expansion, prompt building, Grounding Check, Citation, filing, and the Cannot Confirm gate — is reused unchanged. The only genuinely new code is the dense index plus the RRF merge.

This ships as **RRF-only v1**: RRF is a *recall-union* step (it rescues a relevant Section that one method ranked low — the FM1 keyword-miss case). An optional cross-encoder reranker (the *precision* step for FM2 semantic false positives) is **deferred to a follow-on gated on this phase's eval** (tracked separately), per the roadmap Phase 13 prep note.

## Considered Options

- **Same-corpus fusion over `wiki/` (chosen).** BM25 and dense both retrieve wiki Sections; ids align 1:1; downstream is reused as-is. The dense index is built from `markdown_kb.indexer`'s Section list.
- **Cross-corpus / federated fusion (rejected).** Fuse the *outputs of the two existing stacks* — Wiki-BM25 over `wiki/` plus Vector-RAG-FAISS over `docs/` Chunks. Rejected: it fuses two different corpora at two different granularities, forcing a Chunk→Section aggregation and a cross-corpus id mapping; it breaks downstream reuse and makes the comparison uninterpretable. This is the literal reading of "combine wiki and RAG" and is **not** what Hybrid means here.
- **Char-Chunk granularity for the dense arm (rejected).** Would force a chunk→Section aggregation step before fusion and break 1:1 id alignment. Section granularity keeps fusion trivial and, per the roadmap note, avoids the "wrong retrieval unit" failure (FM3) by construction.
- **Extract the pluggable `Retriever` protocol now (rejected — deferred to #107).** [ADR-0002](0002-two-parallel-retrieval-apps.md) deferred the protocol "until both retrievers work end-to-end"; that precondition is **now met**. We still defer: extracting it would be a cross-cutting refactor touching `markdown_kb` and `vector_rag`, violating the additive promise. Hybrid is added via the existing string→callable dispatch instead. #107 can later land as a pure refactor with Hybrid as its third implementation.
- **Reuse `text-embedding-3-small` vs a different embedding model (chosen: reuse).** Same model the Vector RAG stack uses — keeps cost identical and the three-arm comparison embedding-consistent.
- **RRF-only vs RRF + cross-encoder reranker in this phase (chosen: RRF-only).** Ship fusion first, measure, then decide on the reranker — one variable at a time. The reranker has ready FM2 acceptance cases waiting.

## Consequences

- **Invariant — same-corpus id alignment.** The dense-over-wiki index MUST be built from `markdown_kb`'s filtered Section list so its ids match the BM25 Section ids 1:1. Pointing `vector_rag` at `wiki/` (filename-based ids, no slug) is forbidden — it breaks fusion.
- **Invariant — blessed cross-package reuse of filter/threshold internals.** `hybrid_kb` MUST reuse `markdown_kb`'s `_passes_index_filter` + `_section_lang` (the canonical index filter + language tag) and `vector_rag`'s `_max_rag_distance()` (the calibrated dense gate) rather than re-implementing them — re-implementation would drift the dense arm off the BM25 arm's filtered Section set and break the same-corpus id alignment above. These three are the **only** blessed cross-package `_private` imports; any other reuse of another package's `_private` symbol is a [`CODING_STANDARD.md`](../CODING_STANDARD.md) §2.4 / §11 drift signal until a superseding ADR blesses it.
- **Invariant — Hybrid's pre-LLM gate is a per-arm OR-gate in its own deep module.** Extending the §4.3 *gate-parity* invariant: Hybrid enforces the Cannot Confirm pre-LLM relevance gate inside `hybrid_kb`'s retrieval deep module by evaluating **each arm's native score before fusion** — proceed if the BM25 arm clears `KB_SCORE_THRESHOLD` (per language) **OR** the dense arm clears its eval-calibrated distance gate; otherwise Cannot Confirm (`below_threshold`). OR is required (AND would defeat recall-union). The RRF fused score is **never** used as a relevance threshold — it is not a calibrated relevance magnitude. No new threshold is introduced; the two existing eval-calibrated thresholds are reused.
- **Invariant — additive.** `markdown_kb` and `vector_rag` are not modified. The three interfaces gain a `hybrid` dispatch branch only.
- **Invariant — committed dense seed integrity.** The dense-over-wiki index ships as a committed seed under `.kb/`; a guard test MUST assert the committed seed carries correct metadata and id-alignment with the BM25 wiki index, so a stale seed cannot pass a green fresh-build suite while breaking production (the #307 lesson).
- **The Phase 8 retrieval comparison report is superseded.** Hybrid is added to the comparison as a third arm and **all three arms are re-run** under an upgraded methodology: each arm overfetches a deep candidate pool (target top-50) decoupled from the final cutoff; hit-rate is reported across a cutoff sweep (hit@{1,3,5,10}) plus MRR; a three-way **Cochran's Q** omnibus gates **post-hoc pairwise McNemar** (Wiki↔RAG, Hybrid↔Wiki, Hybrid↔RAG) with Holm correction. The single-cutoff (hit@3) Phase 8 report is replaced; the methodology delta is documented so future readers are not confused by the changed numbers. If the re-run shifts the Wiki-vs-RAG verdict at any cutoff, `README.md` and `why-wiki.md` must be updated.
- **The `Retriever` protocol stays deferred to #107.** ADR-0002's "premature until both work" bar is lifted, but the protocol is still not extracted this phase (see Considered Options). [`CODING_STANDARD.md`](../CODING_STANDARD.md) §2.2 / §10 / §11 are updated to reference this ADR so the reviewer treats a third stack added via dispatch as intentional, not a drift signal.
- **`hybrid_kb` owns one log channel** (`hybrid_kb/log.md` via its own `log_event`) per CODING_STANDARD §5.1, and registers any new `kind` in `log-kinds.md` in the same slice.
- **`hybrid_kb.query()` is a new LLM-facing surface.** It earns one live smoke test (real embedding + real LLM) per CODING_STANDARD §6.4 / ADR-0005's surface enumeration; all fusion / gate / metric / statistics logic is covered deterministically with synthetic inputs.
- Multi-turn Query Rewriting lives in the Gateway ahead of dispatch (ADR-0013) and is stack-agnostic, so Hybrid inherits multi-turn behaviour on the Web path with no new code; CLI/MCP stay single-turn.
