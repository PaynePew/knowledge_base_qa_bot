# failed_grounding pages are quarantined from the retrieval corpus; C3's executable exit is a Routed fix-the-Source flow, and no delete affordance is offered

The fail-soft grounding check (Phase 4 Slice #4) writes a page that fails ingest-time grounding to disk with `status: failed_grounding` plus a `grounding_failure` frontmatter block. The 2026-07-03 grill started from an operator pain ("Re-ingest keeps failing, no repair option") and code exploration surfaced two structural facts that reshaped it:

1. **Machine-verified ungrounded synthesis is being served.** The indexer's status gate is qa-only (`_passes_index_filter`: "non-qa pages pass through"), so a `failed_grounding` entities/concepts page sits in the BM25 corpus, retrievable and citable by `/chat`. The chat-time [[grounding-check]] structurally cannot catch it: it verifies *answer vs cited page*, while the defect is *page vs Source* — exactly the vertical-axis gap CONTEXT.md documents. The live corpus had 6 such pages at grill time, all `claim_unsupported`.
2. **C3's only button is a mechanical no-op for its dominant failure mode.** For `claim_unsupported`, re-ingesting feeds the same LLM the same unchanged Source — the retry fails identically, by construction. The real fix (change what the Source says, or accept the Source cannot support a page) had no affordance at all.

## Decision

1. **Quarantine.** The Section Index excludes any wiki page with `status: failed_grounding` from the retrieval corpus. Pages without a `status` field (legacy) pass through as live, matching the existing "drift state unknown" posture. A page returns to the corpus by the paths that already exist: a re-ingest that passes grounding (status flips to `live`) or a reconcile that clears the failure (`reconcile.py` already pops `grounding_failure`). The committed `.kb/index.json` seed is re-baked with the new filter.
2. **C3's executable exit is a Routed remediation: fix the Source.** Per [[adr-0027]]'s class definition — the missing ingredient (what the Source *should* say) is knowledge only the human can supply, so the affordance navigates, never executes. The Console C3 card routes into the existing pipeline stepper pre-seeded with the finding's context: Source filename, the `unsupported_claims` list, and a `/read/file` view link. The human edits the file locally and the flow closes over existing machinery: Upload (same-name `.md` overwrite — already the behavior) → force re-ingest → re-lint. Zero new endpoints.
3. **The Direct Re-ingest button stays, but `suggested_action` splits by reason.** `verifier_unavailable` → Re-ingest is the correct fix (transient failure); `claim_unsupported` → the text names the unsupported sentences and points at the fix-the-Source route. The undifferentiated retry copy is what produced the "keeps failing" loop.
4. **No delete affordance on C3, and the `DELETE /pages/{slug}` predicate is not widened.** Deleting a failed_grounding page is *unstable*: `source_hashes` (the hash-skip state) lives in the page's own frontmatter, so deletion destroys the skip evidence and any later ingest over `docs/` resurrects the page — same Source, same LLM, same failure. [[adr-0025]]'s full-orphan delete is stable precisely because the Source is gone and ingest can never run again; C3 is the opposite case. Giving up on a Source entirely is expressed by removing the Source file itself (a Source-owner act, deliberately outside this app) — the page then becomes a C11 full orphan and exits through the existing Confirmed delete. The C3 card copy names this path instead of offering a button.
5. **No in-app Source editor.** Editing `docs/` from the synthesis app is the W3 direction [[adr-0003]] rejected; the layered posture (Sources owned outside the curated layer, immutable to the app's write paths except byte-ingress via Upload/Import/Capture) stands.

## Considered Options

- **Keep serving failed_grounding pages, with a warning badge on citations (rejected).** Grounded answers are the project's identity; "this citation is machine-verified unsupported, but here it is anyway" is not a posture a reader can act on. The qa draft gate already established that not-yet-approved content stays out of the corpus; machine-*rejected* content has a strictly stronger claim to exclusion.
- **Extend the ADR-0025 delete predicate to `full-orphan OR failed_grounding` (rejected).** This was the handoff's leaning, but the resurrection instability (decision 4) makes the delete a false exit: the page returns on the next corpus-wide ingest with no record that a human had discarded it. Stabilising it would require a "do not synthesize this Source" suppression state — a new lifecycle with its own consumers (ingest, lint, console) for a case quarantine has already de-fanged.
- **In-app Source editing (`PUT /docs/{filename}` or an editor UI) (rejected).** Reopens ADR-0003's rejected W3 direction and moves Source ownership into the curated layer's app. The Routed flow plus Upload's existing same-name overwrite covers the mechanical need; the judgment stays with the Source owner.
- **Quarantine via a separate directory (move the file out of `entities/`/`concepts/`) (rejected).** Location-based state duplicates what `status` already encodes, breaks the page's identity/inbound links while it is being repaired, and adds a consumer-cascade (lint path expectations, wiki index, ingest overwrite target) for no governance gain.

## Consequences

- The 6 live C3 pages leave the corpus: queries they used to answer become honest Cannot Confirm until their Sources are fixed. This is the intended reading — the pages were serving unsupported claims.
- The C3 display work (Console/CLI render `unsupported_claims`; reason-split `suggested_action`; four-surface sync) and the `_verify_draft` claims-derivation fix are companion tickets, not part of this decision record.
- `_REMEDIATION_TAXONOMY` gains a Routed descriptor for C3 (`route="fix-source"`) alongside its existing Direct Re-ingest — the first finding carrying two remediation classes; surfaces render both per [[adr-0017]] parity.
- Quarantine composes with lint unchanged: C3 keeps firing while the page sits quarantined (lint scans disk, not the index), which is correct — the finding *is* the honest backlog entry.
- **Invariants (tagged for the reviewer, per CODING_STANDARD §2.5).** **Invariant** — the Section Index never admits a page whose `status` is `failed_grounding`; a page with no `status` field is treated as live. **Invariant** — C3's fix-the-Source affordance is Routed: it commits nothing itself; all writes happen in Upload/Ingest under their own rules. **Invariant** — no delete affordance is offered on a C3 finding, and `DELETE /pages/{slug}`'s full-orphan predicate is not widened by this ADR.

## Amendment (2026-07-12, [[adr-0041]])

Decision 4's supporting clause — "Giving up on a Source entirely is expressed by removing the Source file itself (a Source-owner act, deliberately outside this app)" — is **narrowed** by [[adr-0041]]: retire, restore, and rename become governed in-app acts (whole-file atomic moves under audit; bytes never edited), and **byte-purge is the only remaining out-of-app act**. Decision 5 (no in-app Source *editor* — no byte-editing surface) stands unchanged, as does everything else in this ADR. The C3 card copy that names the "remove the Source file" path should point at the in-app Retire act once it ships.

## Amendment (2026-07-20, [[adr-0043]])

Decision 5 is **narrowed** by [[adr-0043]] (issue #632): the C5 fix-source exit gains an **in-modal Source editor** whose staged text uploads through the existing `/upload` byte-ingress under the existing origin-path guards — the human still authors every byte, so "the judgment stays with the Source owner" holds. What this decision rejected and still rejects: app-*authored* Source content, and a general editing surface (`PUT /docs/{filename}` or an always-available editor — the Source browser rail stays read-only). C3's own fix-the-Source banner is unchanged (local edit + Upload); extending the in-modal editor to C3 is a possible follow-up, not decided here.
