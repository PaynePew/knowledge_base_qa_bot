# The Curation gate gains Re-file (C9) and edit-before-promote, both server-re-validated; gates resolve on human surfaces only

Tier B turns the [[remediation]] taxonomy's Gated class ([[adr-0024]]) into working Curation Queue flows. Three decisions from the 2026-07-02 grill share one theme (what the human gate *is* and where it can be resolved), so they are recorded together.

**1. C9 stale-qa is remediated by a single chained operation, `POST /qa/{slug}/refile` (Authored).** A C9 finding means a live Filed Answer's cited entity/concept Sections were re-ingested after it was filed. The fix re-derives the answer, in this internal order — the order is the design:

1. **Re-synthesize:** run the qa page's question through the existing chat pipeline with `wiki/qa/` **excluded from retrieval** for this call. Answers must re-derive from entities/concepts; this mirrors the filing novelty-gate's philosophy (qa-grounded answers are derivative) and dodges the trap where retrieval hits the stale qa page itself and the novelty gate skips filing.
2. **Grounding-check before any write:** if the fresh answer fails [[grounding-check]], **nothing changes** — the old live page keeps serving, the C9 finding stays, the response reports "cannot re-ground". This kills the stranded state (demoted, no replacement, corpus hole for nothing).
3. **Only on pass, demote + overwrite in place:** write the fresh answer onto the **same slug** with `status: draft`, `updated: now`, then one BM25 reindex (the stale answer leaves the corpus). The curator reviews the re-filed draft in the Curation Queue and promotes it live (existing machinery, one more reindex).

The verb is the codebase's own — qa.py's sentinel already says "delete the file to **re-file** fresh"; the operation is the Filed Answer going through filing again. The corpus gap between re-file and promote is deliberate and honest: the old answer was flagged stale, and while the replacement is pending, `/chat` answers from the entities directly. Governance: the compound is **Authored** — the subtractive half (removing known-stale content) behaves like a Direct lifecycle flip; the additive half (LLM synthesis) is what classifies it, and its gate is the promote. No standalone demote endpoint ships: tier B builds only what C9 needs, and ADR-0012's foreseen "deliberate demote flow" is realised inside re-file.

**2. Drafts become editable before promote, via `PUT /qa/{slug}` (draft-only), with a server-side grounding re-check.** The backend's stance already existed (qa.py sentinel: hand-edits are authoritative once live) but the Console offered only Promote/Discard — a curator facing an answer that is 80% right had to publish it verbatim or throw it away. The new endpoint accepts edited question/body for a **draft** page only (refuses `status: live`; live hand-edits remain the documented file-level path), re-runs the LLM-free grounding-check on the edited text against its cited Sections, and rejects with the failure list (422) when the edit breaks grounding. This completes the gate's verb set — **approve / edit-then-approve / discard** — the standard enterprise KB review flow, and re-filed C9 drafts inherit it for free since they land in the same queue.

**3. Gates resolve on human surfaces only.** The Console and CLI are operated by a present human; the MCP surface is driven by an agent. Per ADR-0024's definition (Gated = commits nothing until a *human* approves), a gate exposed as an MCP tool is unverifiable — the server cannot distinguish "the human read the draft and told the agent to promote" from the agent acting alone. So: **MCP sees everything and approves nothing.** MCP lint/queue output gains full visibility (question, draft path, body preview) so the agent can report to its human; gate-resolving operations (promote, discard-of-C8 candidates, re-file, orphan-delete, edit) get no MCP tool. The CLI gains a thin `kb qa` group (`list / show / promote / discard`) wrapping the deep modules — its human is real, so its gate is real. This is the second deliberate asymmetry in [[adr-0017]]'s otherwise symmetric parity (the first is MCP-only Capture), and it points the same way: each surface gets what its operator's hands can honestly do. Agent-as-curator is a real future question, deferred to the Phase 12 (persistent agent) grill — not smuggled in here.

## Considered Options

- **Two-step C9 remediation — a Demote button, then the operator manually re-asks in chat (rejected).** Leaves a stranded trap state (demoted, forgotten, no replacement) and a broken flow (the operator must know to go re-ask the exact question to trigger re-filing).
- **Re-file to a new slug and atomically swap on promote — no corpus gap (rejected).** Requires zombie-slug management plus an atomic two-page lifecycle swap, reinventing exactly the near-duplicate-draft confusion the #341/#342 hardening removed. The gap is small, honest, and self-healing.
- **Pinned re-synthesis from the original citations instead of full re-retrieval (rejected).** More "faithful" but misses newly relevant Sections in the updated entities and requires a bespoke synthesis path; full re-ask with a qa-exclusion filter reuses the chat pipeline wholesale.
- **Publish curator edits without re-grounding — the human is the authority (rejected).** Grounded answers are this project's identity, and no lint check re-grounds qa pages today (C9 checks staleness only), so an ungrounded edit would be invisible drift. The re-check is LLM-free and instant; there is no cost argument for skipping it.
- **Full MCP gate parity (rejected, deferred).** See decision 3 — an unverifiable gate is not a gate. Revisit at Phase 12 with the persistent-agent posture on the table.
- **In-console editing of live pages (rejected).** Scope discipline: the gate flows need draft editing only; live hand-edits keep their documented file-level path.

## Consequences

- Tier B's new-endpoint ledger grows to four, each separately decided: `DELETE /pages/{slug}` ([[adr-0025]]), `POST /qa/promote-batch` (pre-authorized in ADR-0023), and now `PUT /qa/{slug}` + `POST /qa/{slug}/refile`.
- `_REMEDIATION_TAXONOMY` flips C9 from `deferred` to an Authored descriptor carrying the re-file action; the shared taxonomy keeps Browser/CLI/MCP labelling identical ([[adr-0017]]).
- Re-file burns LLM tokens under the daily cap — per-item only, which the Authored class guarantees anyway (Gated never batches).
- Console gate legibility is part of this work's spirit: card copy in operator language (what promoting *does*), and post-promote feedback that closes the loop ("ask X in chat — it now cites this page"). Folded into the Console slices, not a separate decision.
- **Invariants (tagged for the reviewer, per CODING_STANDARD §2.5).** **Invariant** — a failed re-ground during re-file writes nothing: no demote, no draft, no reindex. **Invariant** — un-approved synthesis never enters the corpus: re-file and edit stage drafts; only promote commits ([[adr-0020]]). **Invariant** — `PUT /qa/{slug}` accepts drafts only and re-runs grounding server-side; it never touches a live page. **Invariant** — no MCP tool resolves a Gated remediation (promote, discard, re-file, edit, orphan-delete are Console/CLI only).
