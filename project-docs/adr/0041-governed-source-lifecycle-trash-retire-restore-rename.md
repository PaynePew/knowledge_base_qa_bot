# Source lifecycle becomes a governed in-app surface: retire is an atomic move into a timestamped trash outside docs/, restore reverses it, rename mechanically re-points derived frontmatter; byte-purge is the only remaining out-of-app act

A deployed operator (browser only, no filesystem access) had no in-app way to rename, retire, or remove a Source: [[adr-0029]] defined Source removal as an out-of-app act (`rm` the file), and [[adr-0036]] deliberately locked fix-source to *content overwrite, same filename*, deferring rename/retire to "its own decision" — this ADR (2026-07-12 grill, issue #530). The design premise is a reframe of [[adr-0003]]'s immutability posture: the load-bearing intent is **"the app never silently or irreversibly mutates a Source"**, not "the app has zero Source-write surface" (Upload / Import / Capture already write Sources). Governed, reversible, audited whole-file *moves* are lawful; byte edits are not. The 2026-07-07 industry survey (MediaWiki move-leaves-redirect, Confluence/Notion trash-with-retention, docs-as-code git-owned deletes, event-sourcing tombstones) converges on the same posture: never unlogged hard-delete of a canonical record, and *move is a much lower bar than delete*.

Three code facts fixed the mechanism during the grill: (1) derived pages reference their Sources by **bare filename** (`sources: <filename>#<heading>`), re-resolved on demand via `docs_dir.glob("**/<filename>")` — and pathlib's `**` descends dot-directories, so a trash *inside* `docs/` would need a never-miss exclusion in 3+ scanners (`upload._resolve_overwrite_target`, `ingest._resolve_single_source_pairs`, `lint._resolve_c3_source_path`) or retired files resurrect on the next corpus-wide ingest — the exact instability [[adr-0029]] decision 4 names. (2) `_passes_index_filter` quarantines only `failed_grounding` for entity/concept pages, so a "demote derived pages on retire" cascade would not even exit the corpus without a new status value plus an index-filter semantics change. (3) [[adr-0025]]'s delete gate recomputes the full-orphan predicate under lock at delete time, so retire/restore races against a stale C11 report are already handled (restore first → the old report's delete click 409s).

## Decision

1. **Action set: v1 builds retire, restore, and rename.** Supersede (content fix) is already shipped as [[adr-0036]] fix-source and is only referenced. **Byte-purge is not built**: [[adr-0029]]'s "Source removal is only an out-of-app act" clause is *narrowed, not repealed* — **byte-purge (true erasure) is the only remaining out-of-app act** (`rm` / `git rm`). The single-curator model stays; no RBAC (the survey's role tiers are multi-user machinery; a single curator needs soft-delete plus an audit trail, not privilege levels).

2. **Retire (Confirmed) = one atomic whole-file move to `<kb-root>/.trash/<UTC-timestamp>/docs/<original-relpath>`.** The trash lives **outside `docs/`**, so every Source scanner is structurally blind to it — no exclusion lists to maintain, no resurrection path, by construction. Source bytes are never touched (move ≠ mutate). The timestamped act-folder gives collision-free repeated retires of the same relpath and a `grep`-able physical audit sequence. The confirmation dialog is an **impact preview** computed server-side with lint's own citation resolution: which derived pages become full orphans (deletable) and which partial. The retire response routes the curator to the resulting C11 findings.

3. **No cascade onto wiki pages — the vacuum window is accepted and curator-held.** Between retire and the C11 Confirmed delete, derived pages keep serving; the same person in the same flow closes the window, and an abandoned flow leaves the C11 finding as the honest backlog entry ([[adr-0029]] posture: lint scans disk, the finding keeps firing). Rationale: a cascade needs a new page status plus an `_passes_index_filter` extension (a prod-fragile retrieval-semantics change) and reverse bookkeeping on restore ("which pages did *this* retire demote"), entangling the Source lifecycle with the page lifecycle — the "second lifecycle to govern" [[adr-0025]] warned about. Page corpus-exit keeps its single owner: the existing C11 → Confirmed delete machinery, unchanged. Content that must leave *immediately* has the right tools already: fix-source (change what it says) or retire-then-delete-now (the delete button is in the retire response).

4. **Restore (Direct) = the atomic inverse move**, keyed by trash entry (timestamp + relpath). Two refusal guards, sharing [[adr-0036]] §6's *refuse, never fall back* posture: the original relpath is now **occupied** (a same-name Source was uploaded after the retire), or the basename **collides** elsewhere under `docs/` (restoring would mint the `ambiguous_source` state ADR-0036 §6 exists to prevent). Never overwrite, never auto-suffix — silently minting a new canonical name is the exact ungoverned mutation this ADR forbids. Restore does zero page bookkeeping: extant orphan pages are cleared by the next lint recompute; already-deleted pages re-synthesize on the next ingest because their `source_hashes` skip-state died with the page ([[adr-0029]] decision 4, now working *for* us).

5. **Rename (Direct) = atomic same-directory basename move + mechanical re-point + one reindex.** Every derived page's `sources` entries and `source_hashes` keys are rewritten `old.md → new.md` in the same act, then one `build_index()` (the delete/promote pattern). **No LLM**: re-synthesizing unchanged bytes costs money and invites synthesis drift for zero information. **No alias is created** — the issue's proposed "alias old-slug → new page" row was a category error: [[adr-0030]] aliases map *page slugs*, and page slugs are minted from the Source *outline* ([[adr-0027]]), which a file rename does not change; no page identity moves, so there is nothing to redirect. The mechanical re-point is [[adr-0025]]'s own C11-partial remediation ("fix the renamed citation in frontmatter") promoted from a hand edit to one atomic act. Guards: new name passes the upload safe-basename rules; refuse if the new basename already exists anywhere under `docs/`; refuse when the origin resolves `ambiguous`/`missing`.

6. **One shared validator** backs all three acts (safe-basename ∧ target-occupancy ∧ corpus-wide basename-collision ∧ origin resolution) — the same guard vocabulary as fix-source, in one place, so no lifecycle act can disagree with another about what a lawful destination is.

7. **Surfaces: Console primary, CLI parity, MCP visibility-only.** Rename/retire mount on the existing browser-rail file rows (no new Sources management page); the rail gains a Trash section (timestamp, original relpath, preview, Restore). CLI gains `kb source rename|retire|restore|trash` (the `kb alias add` precedent). MCP sees lifecycle state and writes none of it — retiring or renaming a Source is a semantic judgment about the corpus, exactly the class [[adr-0026]] keeps off agent hands until Phase 12.

8. **Audit = the trash tree + append-only log lines; no manifest.** The timestamped tree already answers what/when/from-where; each act additionally logs one bounded `log_event` line (act, relpath, impact counts — §5.1 own-channel, §5.3 no full content) from the source-lifecycle module, which lives in `markdown_kb/app/` beside upload/ingest/lint. A separate lifecycle manifest is rejected: hand-authored authoritative state in `.kb/` is [[adr-0030]]'s category error, a manifest under `docs/` reopens the dot-file pollution problem, and every scanner would have to learn to consult it.

## Considered Options

- **`docs/.trash/` inside the corpus tree (rejected).** Needs a never-miss exclusion in every present and future `docs/` scanner (pathlib `**` descends dot-dirs), or a preliminary "one shared docs-scan helper" refactor that doubles this ADR's diff; one missed glob = `ambiguous_source` or silent resurrection. The out-of-tree trash gets the same guarantee for free, structurally.
- **Sidecar lifecycle manifest, file stays in place (rejected).** Same consumer-cascade as the exclusion approach with none of the physical invisibility; and see decision 8 for where such a file could even live.
- **`status: retired` marker written into the Source file (rejected).** Sources carry no frontmatter (seed files are frontmatter-free; the importer writes provenance only); writing a marker mutates canonical bytes — the one thing the reframe forbids the app to do.
- **Cascade quarantine of derived pages on retire (rejected).** Zero vacuum window, but: new status value + index-filter extension (prod-fragile), reverse bookkeeping on restore, and permanent entanglement of two lifecycles. See decision 3.
- **Auto-delete full orphans as part of retire (rejected).** Confirmed Remediations never batch and never piggyback ([[adr-0024]]); the impact preview + routed C11 exit gives the same closure with the human gate intact.
- **Rename as copy + tombstone-old + alias redirect (the issue's original row — rejected).** Category error at the file layer (see decision 5), plus it leaves two records of one canonical file and churns every derived page through orphan-and-remint for a no-op content change.
- **In-app purge with retention expiry (rejected).** The public box is ephemeral ([[adr-0021]] resets), repo-backed deployments purge via git, and a single-curator model has no privilege tier to hang a "highest bar" on. Purge stays out-of-app; revisit only with multi-user RBAC.

## Consequences

- [[adr-0029]] is **amended** (see the Amendment appended there): its decision-5 clause narrows to "byte-purge is the only out-of-app act"; its core (no in-app Source *editor*, no byte-editing surface) stands.
- [[adr-0025]] is **untouched**: its rejection of a trash layer was scoped to *derived pages*, which re-ingest can regenerate from Sources; a Source is canonical bytes nothing can regenerate, so reversibility must live at this layer. The two postures compose: soft-demote for the wiki layer, trash-move for the Source layer.
- [[adr-0003]] is **deepened, not edited**: "immutable to the app's write paths" is henceforth read as "never silently or irreversibly mutated"; this ADR is the reference for that reading.
- CONTEXT.md gains the Source-lifecycle vocabulary (Retire / Restore / Rename / Source Trash), and the "Section ID is not stable across renames" flagged ambiguity is **partially resolved**: a governed in-app rename re-points derived references in the same act; heading-edit instability and out-of-app renames remain as flagged.
- **New-endpoint ledger**: `GET /sources/{relpath}/impact`, `POST /sources/retire`, `POST /sources/restore`, `POST /sources/rename`, `GET /sources/trash`; `read._WHITELIST_ROOTS` gains a read-only `.trash` root for pre-restore inspection.
- Only rename triggers a reindex (frontmatter changed). Retire/restore never touch wiki pages, hence never reindex; corpus membership changes only through the existing C11 delete (which owns its own reindex).
- Implementation-time check, no machinery: a heading-less Source's fallback Section id derives from the filename; after a rename its derived page keeps the old-name slug — a page slug is an identifier, and a curator who cares can assign an [[adr-0030]] alias by hand.
- CODING_STANDARD §10 gains the **Trash-move (Source tombstone)** pattern row; the freshness stamp bumps to this ADR.
- Slices: **S1** core (trash move/restore + shared validator + the four endpoints + log + tests), **S2** rename re-point (prod-fragile: derived-page frontmatter shape + reindex), **S3** surfaces (browser rail + CLI parity + MCP visibility). S1 → S2, S1 → S3.
- **Invariants (tagged for the reviewer, per CODING_STANDARD §2.5).** **Invariant** — no in-app act ever mutates Source bytes: lifecycle acts are whole-file atomic moves only. **Invariant** — every lifecycle act refuses on ambiguous/missing origin, occupied target, or basename collision — never falls back, never silently mints a new name. **Invariant** — retire and restore never modify wiki pages; a page's corpus exit remains solely the existing C11 Confirmed delete. **Invariant** — nothing under `.trash/` is ever ingested, indexed, or resolved as a Source; the trash lives outside every `docs/` scanner's tree. **Invariant** — MCP exposes no Source-lifecycle write verb.
