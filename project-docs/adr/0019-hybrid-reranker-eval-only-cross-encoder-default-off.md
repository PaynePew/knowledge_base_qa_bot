# Hybrid's reranker is a default-off cross-encoder, measured eval-only and never loaded on the VPS tenant

Phase 13 [ADR-0018](0018-hybrid-retrieval-third-stack-rrf-over-wiki.md) shipped Hybrid Retrieval as **RRF-only v1** and explicitly deferred the cross-encoder **reranker** — the *precision* step for FM2 semantic false positives — to a follow-on "gated on this phase's eval" (#310). This ADR records that follow-on. We add a reranker to **Stack C only**, behind a flag (`KB_HYBRID_RERANK`) that is **default-off everywhere**, scored with a real local cross-encoder (`bge-reranker-v2-m3`, multilingual incl. Chinese) **inside the paraphrase-comparison eval only**. The heavy dependencies (`sentence-transformers` + `torch` + the ~2.3 GB model) are an **optional, lazily-imported** extra: the slim production image neither installs nor loads them, and the live `ask-wiki-rag` VPS tenant **never** runs the reranker. Whether it ever ships live is a separate, post-eval decision.

RRF is a recall-union step that *rescues* a relevant Section one method ranked low; the reranker is the orthogonal *precision* step that re-scores the fused pool and demotes a high-fused-rank-but-semantically-wrong Section. They compose — the reranker does **not** replace RRF. At this corpus (~51 Sections) `candidate_depth=50` ≈ the whole corpus, so gold is almost always already in the fused pool; the probe miss is therefore a *ranking* failure, which is exactly what a reranker fixes.

## Considered Options

- **Eval-only local cross-encoder (chosen).** Measure the reranker's ceiling with the strongest open multilingual cross-encoder on the dev box; never load it in production. The eval question is "is reranking worth doing here *at all*" — answered most honestly with the best available scorer, offline, reproducible, with **zero recurring cost** and no new vendor. The ~2.3 GB model is a one-time dev-machine cost behind an optional dependency; it cannot reach prod.
- **Rerank API — Cohere `rerank-3.5` / Voyage `rerank-2.5` (rejected for v1).** CJK-capable and the *only* option that could realistically run against the bulkheaded tenant (compute offloaded), so its measured latency would transfer 1:1 to a prod decision. Rejected for v1 because it commits to a new vendor + API key + per-query billing before we know reranking helps at all, and it makes the eval non-deterministic. This is the natural backend to revisit **if** the eval gate passes and a live rollout is wanted.
- **Tiny local CJK cross-encoder shipped live (rejected).** Even a small multilingual cross-encoder (`mmarco-mMiniLMv2`, ~470 MB) plus `torch` will not fit the per-tenant bulkhead (`mem_limit: 512m`) on a 2 GB / 2 vCPU box shared by ~3 tenants without starving co-tenants — and a quantized model under-measures the ceiling the eval exists to establish.
- **LLM-as-reranker reusing `gpt-4o-mini` (rejected for v1).** Adds no new dependency, but it is non-deterministic, bills per query, and measures *LLM listwise reranking*, not the dedicated **cross-encoder** ADR-0018 named. The clean, reproducible ceiling measurement wants a cross-encoder.
- **Fold the new arm into a 4-way Cochran's Q omnibus (rejected).** The rerank question is a *within-Hybrid* paired contrast (C vs C+rerank), so we report it as a focused paired McNemar and keep the existing three-arm (Wiki/RAG/Hybrid) omnibus intact rather than diluting that narrative into a symmetric 4-way test.

## Consequences

- **Placement (extends ADR-0018's flow).** `retrieve_and_gate` keeps `overfetch → OR-gate → RRF`, but RRF now emits a **deep fused pool** (`rerank_depth`, default **20**); when the flag is on, the reranker reorders that pool and the result is truncated to `top_k=3`. The **Cannot-Confirm OR-gate is unchanged** — it stays on each arm's *native pre-fusion* score (BM25 top score / dense min distance) and **never sees the reranker**.
- **No new committed seed.** The reranker scores at query time, so there is **no** `.kb/` artifact and no re-bake (unlike the dense seed). This sidesteps the #307 stale-seed class of bug for this feature by construction.
- **Flag shape.** `KB_HYBRID_RERANK` (default off) is owned by `hybrid_kb` and read *through* the module at call time so tests monkeypatch it, mirroring `KB_SCORE_THRESHOLD` / `KB_RAG_DISTANCE_THRESHOLD`. The reranker lives in a new deep module `hybrid_kb/app/rerank.py` whose single network/model seam (`get_cross_encoder()`, a lazy singleton) is the only thing hermetic tests swap.
- **Production never loads it.** `sentence-transformers` / `torch` / the model are an **optional** dependency group, imported **lazily** inside `rerank.py`. Importing `hybrid_kb` with the flag off never imports them, so the prod image stays slim and a missing dependency is not an error in the default (off) path. If the flag were ever flipped on in an environment without the extra, the call fails fast with a clear message rather than degrading silently.
- **Bilingual.** `bge-reranker-v2-m3` is multilingual (incl. Chinese), matching the corpus; no per-language reranker routing is introduced (the per-arm language filter already happened upstream of fusion).
- **Reranked score is not surfaced and not a threshold.** Extending ADR-0018's "the RRF fused score is not a calibrated relevance magnitude": the cross-encoder's score likewise is **never** exposed as a citation score (PROMPT.md Q3) and **never** used as a relevance gate. It only reorders.
- **Additive, Hybrid-only.** `markdown_kb` and `vector_rag` are untouched; the Wiki and RAG arms do not gain a reranker in v1. No new cross-package `_private` import (the module reuses only the already-blessed `Section`).
- **Eval: a focused 4th arm.** `eval/paraphrase_comparison` adds **Stack C+rerank** — a 4th column in the cutoff sweep + per-type tables, plus a dedicated **paired McNemar (Hybrid vs Hybrid+rerank)** on the same Paraphrases, the probe deltas, and the **measured (dev-box) added latency**. The existing three-arm Cochran's Q omnibus is preserved. Offline `--fake-embeddings` runs still route to `report.offline-tracer.md` (#328); only a real-embedding run writes the canonical `report.md`.
- **Gate for any future live rollout:** probe improvement **AND** no Core hit@3 regression. The dev-box latency is recorded descriptively, not as a blocker (v1 ships off regardless). Flipping the default on — and choosing a deployable backend (the rerank API) — is a separate decision against these numbers, and would update README / why-wiki to a four-arm story.
- **The constraint is a per-tenant bulkhead, not the box size.** `docker-compose.prod.yml` caps *this* tenant at `mem_limit: 512m` / `cpus: "0.5"` as a deliberate bulkhead so one tenant cannot starve the shared 2 GB / 2 vCPU VPS that fronts ~3 tenants behind the shared `edge` Caddy. The reranker is excluded from prod because of that bulkhead (and co-tenant fairness), not because the physical host is tiny.
