# The Hybrid stack's dense arm is refreshed by an operator-triggered full re-embed (`POST /hybrid/index`); promote auto-rebuilds only BM25, and the BM25-auto / dense-manual asymmetry is deliberate

The Hybrid Retrieval stack ([ADR-0018](0018-hybrid-retrieval-third-stack-rrf-over-wiki.md)) fuses a BM25 arm and a dense-vector arm over the curated `wiki/` Section corpus with RRF. The BM25 arm *is* `markdown_kb`'s Section Index, which `promote` already rebuilds in full ([ADR-0020](0020-filed-answer-loop-validated-write-back.md)); the dense arm is a committed FAISS seed under `.kb/hybrid_dense/` that, until now, had **no production rebuild path** — `hybrid_kb` exposes no endpoint and only tests / `eval/` call `dense_index.build_index()`, so the dense arm was frozen on the seed for all post-seed content (a Phase 13 gap surfaced by the #341/#342 Q3 grill and deferred from ADR-0020, tracked as #348). This ADR records how the dense arm is refreshed in production and **why its refresh is governed differently from BM25's**.

Because `reciprocal_rank_fusion` keys by `section.id`, a page present in only the BM25 arm is still served (the dense arm merely adds no recall boost). A stale dense arm is therefore **runtime-safe — it degrades recall, not correctness.** That tolerance is what lets the dense refresh be a deliberate, occasional operator action rather than an automatic per-write side effect.

## Considered Options

- **Operator-triggered full re-embed via a gateway route (chosen).** Add `POST /hybrid/index` — a gateway-owned route that calls `hybrid_kb.app.dense_index.build_index()` in-process, re-embedding every wiki Section and persisting the FAISS seed. `hybrid_kb` stays library-only (no FastAPI app), consistent with how `stack=hybrid` chat already dispatches through a direct `hybrid_kb.query` import; the path name mirrors `POST /rag/index`. A full re-embed is **trivially correct** — it restores the ADR-0018 1:1 dense↔BM25 id alignment by construction and matches disk with no incremental-patch drift surface — and is the symmetric choice to ADR-0020's full BM25 rebuild.
- **Incremental FAISS add on promote/ingest (deferred, not rejected).** A single-embedding `FAISS.add` on promote/ingest (and a remove on discard) would keep the two arms aligned automatically with no operator step. Deferred because it adds per-event embedding cost and a precise add/remove/edit bookkeeping surface that can drift the dense id set off the BM25 Section set — the very invariant ADR-0018 protects. Revisit if the manual cadence proves tedious or near-real-time dense freshness is required; at that point a backend with native upsert is the better move than a hand-rolled FAISS patcher (echoing ADR-0020's incremental-BM25 reasoning).
- **Auto-rebuild the dense arm on promote (rejected).** Symmetric with BM25's auto-reindex, but a full wiki re-embed on **every** promote/discard has real embedding cost and seconds of latency (unlike BM25's sub-second rebuild), and would burn the cost cap the public demo runs under ([ADR-0021](0021-public-demo-ephemeral-and-legible.md)). RRF tolerance makes an automatic refresh unnecessary for correctness. Hence the **BM25-auto / dense-manual asymmetry** — the surprising part this ADR exists to explain.
- **Give `hybrid_kb` its own mounted FastAPI app (rejected).** Structurally identical to how `vector_rag` is mounted at `/rag`, but it adds a service layer to a package ADR-0018 deliberately kept **library-only and additive**; the gateway-owned route reuses the existing dispatch shape and touches less.
- **Serving-time guard that drops stale dense-only ids (shipped in #355, correcting this ADR).** Between rebuilds the dense arm can return a Section for a page `ingest.delete_orphans` has discarded from the live wiki — a dense-only dead id — not an edited page: RRF already resolves the edit case correctly, since `reciprocal_rank_fusion`'s `section_by_id.setdefault` keeps the live BM25 Section object for any id present in both arms. #355 added a serving-time membership filter in `hybrid_kb.app.retrieval.retrieve_and_gate` that drops any dense hit whose id is absent from the full live BM25 Section-id set, applied before the OR-gate and RRF, closing the orphan-case leak; the manual Dense rebuild remains the remedy for the (harmless) missing-new-pages direction.

## Consequences

- **`POST /hybrid/index` is a heavy admin path.** It joins `gateway/app/budget.py`'s `_COST_ESTIMATES` at an estimate on par with `/rag/index` (a whole-corpus re-embed) and `middleware.py`'s `ADMIN_PATHS` (admin semaphore + per-UTC-day cost cap + admin-token kill-switch), so it is cost-capped on the no-auth public demo per ADR-0021 — exposed and legible, not hidden, but ephemeral and bounded.
- **The Operator Console gains a "Hybrid dense rebuild" action** mirroring the existing "RAG rebuild" parallel track, with ADR-0021 always-visible legibility copy and the #347 static-`el()` / no-new-`innerHTML` invariant. Its copy must say it refreshes **only the dense arm** (the BM25 arm already updates on promote).
- **The dense arm lags between rebuilds — in one direction only (corrected by #355).** It misses new promoted/ingested pages (no dense recall boost) — a runtime-safe, RRF-tolerated gap remedied by the next Dense rebuild. This ADR originally also described a second direction — "may surface edited/discarded seed pages as stale text" — but that overstated the leak: the **edit** case (a re-ingested page keeping its id) never reaches serving, because `reciprocal_rank_fusion`'s `section_by_id.setdefault` always keeps the live BM25 Section object for a shared id. The actually-reachable case was the ingest **orphan-cleanup dead id** (`delete_orphans` unlinking a page from the live wiki while the frozen dense seed still embeds it) — #355 closed it with a serving-time membership filter in `retrieve_and_gate` that drops any dense hit absent from the live BM25 Section-id set before the OR-gate and RRF. `test_committed_seed_guard.py` still catches dev-time seed drift independently.
- **Dev-time seed maintenance is unchanged.** The committed `.kb/hybrid_dense/` seed is still re-baked and force-committed (`git add -f`) when `wiki/` changes in the repo; #348 adds the *runtime* rebuild path, not a change to how the seed is produced for the build.
- **Implementation** is a single follow-up slice (the gateway route + budget/middleware wiring + the console action + tests); this ADR records the design from the 2026-06-30 grill.
