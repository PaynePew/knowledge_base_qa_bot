# Grounding verdicts become reproducible: every answer-chain LLM call (rewrite, draft, verify) pins a fixed seed on top of temperature=0, and arithmetic derivation is named an unsupported inference class in the verifier prompt

Re-asking the same question flips between a grounded answer and Cannot Confirm across runs, and one accepted draft carried an **arithmetic-inference claim** — a number combined from source values rather than present in any Section (issue #572, 2026-07-12 grill). PR #283 pinned the rewrite, draft, and verify LLMs to `temperature=0` for exactly this flip; [[adr-0038]] then found that temperature alone does not hold — the C5 judge still flipped verdicts on byte-identical inputs until it gained a fixed `seed` (`_C5_LLM_SEED = 7`, `lint.py`). The serving chain never received the same treatment: `gateway/app/query_rewriting.py:64` (rewriter), `markdown_kb/app/retrieval.py:102` and `:115` (drafter + retry drafter), and `markdown_kb/app/grounding.py:309` (verifier) are all `temperature=0` **without a seed** — and the verifier is additionally constructed per-call inside `verify()` rather than through the lazy-singleton getter every other LLM site uses (§2.7).

The flip has two distinct sources, and they need different fixes. (a) **Sampling nondeterminism**: OpenAI backends do not guarantee identical output at `temperature=0`; a seed narrows this (best-effort — [[adr-0038]]'s caveat stands). (b) **Specification ambiguity**: `VERIFIER_SYSTEM_PROMPT` forbids "logical inference beyond what is written" but never names arithmetic, so a claim like "up to 12 days" derived from "7–10 business days" plus "2 days processing" sits exactly on the rule boundary — the verdict is decided by sampling luck, not by the rule. Seeding alone would freeze (b) to an *arbitrary* verdict, including (as observed in #572) the wrong one. The rule fix makes the correct verdict deterministic **by specification**; the seed makes the whole chain deterministic **by sampling**. Both are required: an unseeded drafter re-words the draft on every re-ask, handing the verifier legitimately different claims, so no verifier-side fix alone can stop the user-visible flip.

## Decision

1. **Fixed seed on all four answer-chain LLM constructions** — rewriter, drafter, retry drafter, grounding verifier — extending [[adr-0038]]'s C5 pattern to the serving chain. Per-module private constants (no cross-package shared constant; the gateway and markdown_kb modules stay decoupled), value `7`, the repo's convention since ADR-0038. The best-effort caveat is documented at each site: OpenAI's seed is not a hard guarantee — it cuts the flap, it does not abolish it (`system_fingerprint` drift across backend updates remains).

2. **The verifier prompt names arithmetic derivation as an unsupported inference class.** Any claim whose value is produced by computing over the content of CITED_SECTIONS — sums, differences, ranges, unit conversions, date arithmetic, counts, aggregations — is *not* explicitly supported, regardless of how plausible the computation is. Paraphrase remains acceptable; computation is inference. This is [[adr-0001]]'s strict-grounding posture applied to the observed boundary case: an honest Cannot Confirm beats a confident derived claim. The #572 accepted-arithmetic case becomes deterministically rejected.

3. **The verifier construction hoists into a lazy-singleton getter** (§2.7), matching every other LLM call site: tests pin the seed by asserting the getter's constructor kwargs through the established `monkeypatch` seam (§6.3) instead of patching module internals, and the client stops being re-constructed on every `verify()` call (the #483 singleton discipline).

## Considered Options

- **Quorum judge — 3 verifier calls, majority verdict (rejected).** Triples cost and latency inside the verifier's 5-second budget on the hot serving path; seeding buys most of the same stability at zero marginal cost. Revisit only with eval evidence of residual flapping *after* the seed lands.
- **Serving-side verdict cache keyed by (draft, evidence) hashes (rejected).** The C5 verdict cache (frozen-42 pattern) works because batch lint re-judges *byte-identical* page bodies. In serving, the cache key only repeats once the upstream chain is already deterministic — at which point the cache stores no new signal — and it adds unbounded per-query state on an open box.
- **Drafter-side rule — "do not assert derived arithmetic" (deferred, eval-gated).** Would lower the Cannot Confirm rate by having drafts quote source values instead of computing over them, but it is a synthesis-quality lever, not an enforcement point. Enforcement stays at the judge — the keystone posture [[adr-0040]] established. Open as a follow-up only if eval shows the Cannot Confirm rate rising after this ADR.
- **Live regression test for the arithmetic rule (rejected).** A prompt-rule behavior is eval territory (`negative_case` / `paraphrase_comparison` arms), not a live test; the one-live-test-per-surface budget (§6.4) is untouched.

## Consequences

- Implementation is **one slice on issue #572**, prod-fragile (grounding semantics) → Rung 2: seed constants at the four sites, verifier getter hoist, the prompt rule, and tests that pin constructor kwargs through the getter seam. No index or seed re-bake — nothing in the committed artifacts encodes verifier sampling.
- Tests assert constructor kwargs and outcome shape only — never LLM output text (§6.2). The prompt-rule's behavioral effect is measured by the eval arms, not asserted in pytest.
- **Expected user-visible shift**: drafts containing derived arithmetic now consistently return Cannot Confirm instead of intermittently passing. That is the intended honesty, not a regression; if eval shows the Cannot Confirm rate climbing, the remedy is the deferred drafter-side rule, never a loosened judge.
- Batch ingest-side LLM calls (`templates.py`, `structure_enrichment.py`) are **out of scope**: their outputs are baked into committed, eval'd artifacts, and transcription nondeterminism is a known separate surface (#508 gotcha).
- CODING_STANDARD §10 gains the **Seeded judge (deterministic LLM call)** pattern row; the freshness stamp bumps to this ADR.
- **Invariants (tagged for the reviewer, per CODING_STANDARD §2.5).** **Invariant** — every serving-chain LLM call is constructed with `temperature=0` AND a fixed seed; a new serving-chain call site must adopt both pins or carry an ADR saying why not. **Invariant** — the grounding verifier accepts paraphrase only; a named inference class (arithmetic derivation) is unsupported regardless of plausibility. **Invariant** — determinism pins never widen to batch ingest surfaces without their own decision record.
