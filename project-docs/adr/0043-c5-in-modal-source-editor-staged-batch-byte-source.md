# The C5 fix-source exit gains an in-modal Source editor; staged corrections are an alternate byte-source for the same batch Upload

Issue #632 (operator report, live console). [ADR-0036](0036-c5-source-rooted-contradiction-routed-fix-source.md) shipped the source-rooted C5 exit as: "Fix this Source" silently accumulates the target into a batch panel on the main page, the curator edits the file **locally**, then re-uploads it through a file picker (decisions 3–5), with decision 4 explicitly rejecting an in-app Source editor. Live use showed two legibility failures:

1. The batch panel lives **behind the modal overlay**, and the modal gives zero feedback on click — "Fix this Source" reads as a dead button.
2. In the Wiki comparison view, a not-converged pair shows an editable draft with a permanently disabled Apply ([ADR-0038](0038-c5-source-rooted-detected-by-convergence-rejudge.md)'s gate, correct) but **no visible reason** — the sources-disagree note rendered only in the Source view.

And one friction failure: for the actual demo-corpus case (add or remove a single "30 天" clause), download-edit-reupload is a hostile round-trip for a one-line correction.

## Why superseding "no in-app editor" is now safe

ADR-0036 decision 4 leaned on [ADR-0029](0029-failed-grounding-quarantine-routed-source-fix.md) decision 5 / [ADR-0003](0003-w2-layered-wiki-target-claude-obsidian.md): Sources are owned outside the curated layer, immutable to the app's write paths **except byte-ingress via Upload**. That doctrine has already been narrowed twice: ADR-0036 decision 6 added the destination-aware in-place Source **overwrite** (through Upload), and [ADR-0041](0041-governed-source-lifecycle-trash-retire-restore-rename.md) gave the app a governed Source **lifecycle** surface (trash/retire/restore/rename). An in-modal editor changes none of the load-bearing properties: the **human still authors every byte** (typed instead of picked), the bytes still enter through the **same `/upload` ingress**, and the server-side origin-path guards (ADR-0036 decision 6, unchanged) still decide whether the overwrite is legal. What ADR-0029 actually rejected — the app *authoring* Source content, or a general `PUT /docs/{filename}` editing surface — stays rejected: the editor exists only inside the C5 fix-source exit, and the Source browser rail remains read-only (lifecycle ops only).

## Decision

1. **"Fix this Source" opens an inline editor in the Reconcile modal's Source comparison card** (supersedes ADR-0036 decision 4). The editor loads the **whole file** read-only via the existing `GET /read/file` convention (`section.source_path` verbatim — no client-side path or section reconstruction, §12.5). ADR-0036 decision 3's "the edit entry point never fetches anything itself" *wording* is superseded; its intent — a click on the entry point never mutates and never launches a pipeline — is preserved: the only request the editor makes is the read-only load.
2. **"Stage correction" is pure client state.** It stores the corrected full-file text on the existing pending-target list (`addC5FixSourceTarget(sourcePath, content)`); re-staging replaces the staged text. No network write happens at stage time.
3. **Every byte-source converges on ONE shared pipeline** (`runC5FixSourcePipeline`): sequential per-target `POST /upload` — each with its own resolved `overwrite_relpath` (issue #533) and the target's own filename, so a staged Blob and a picked File upload identically — then **one** force re-ingest over the entries' filenames, then **one** deep-audit. ADR-0036 decision 5's batch shape is unchanged. An in-modal staged-upload bar (foot of the Source view, hidden while nothing is staged) runs it and closes the modal on success; the curator lands on the re-rendered Lint card.
4. **The batch panel and its file-picker path survive** as the multi-finding accumulation surface and the local-editing escape hatch (issue #544's duplicate-basename claim-matching unchanged). Targets carrying a staged correction are badged and can be uploaded from the panel without a picker; a picked file matching a staged target's basename overrides the staged text for that run.
5. **The sources-disagree note renders in the Wiki comparison view too** when the pair is not converged — the disabled Apply is explained at the point of use, and the note's text now names the in-modal path ("Fix this Source → stage → upload") instead of "upload it below".
6. **Whole-file editing only.** The textarea holds the entire file; the client never splices sections (§12.5). The server-side overwrite guards (existing-Source-only, resolved origin relpath, refuse ambiguous/missing — ADR-0036 decision 6) are untouched.

## Consequences

- ADR-0036 decision 4 is superseded (amendment appended there); ADR-0029 decision 5's rejection is narrowed to app-*authored* Source content and general editing surfaces (amendment appended there). The Apply gate (ADR-0038) is untouched.
- The curator can now resolve a source-rooted C5 finding end-to-end without leaving the modal: open Source comparison → Fix this Source → edit → Stage → Upload staged & re-verify.
- A curator who stages corrections across SEVERAL findings before uploading still gets one re-ingest + one deep-audit for the whole set (the panel's staged-upload button); per-pair immediate upload from the modal is the curator's choice, not a forced stepper.
- **Invariants (tagged for the reviewer, per CODING_STANDARD §2.5).** **Invariant** — staging never issues a request; only an explicit upload run does. **Invariant** — one upload run = one force re-ingest + one deep-audit, whichever entry point started it. **Invariant** — the editor loads and uploads the whole file at the target's resolved `source_path` verbatim; the client never reconstructs paths or splices sections. **Invariant** — the in-app editor exists only inside the C5 fix-source exit; the Source browser rail stays read-only (ADR-0041 lifecycle ops only).
